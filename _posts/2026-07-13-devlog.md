## 종합 랭킹, 결국 포기했다

처음에는 국내(네이버)랑 해외(WATT/GDELT)를 합쳐서 하나의 종합 랭킹을 만들려고 했다.

```python
# scorer.py
def split_domestic_international(articles: list[dict]) -> tuple[list[dict], list[dict]]:
    """
    국내/해외 개별 집계 축 분리. 네이버 = 국내, WATT/GDELT = 해외
    """
    domestic = [a for a in articles if a.get("source") == "네이버"]
    international = [a for a in articles if a.get("source") != "네이버"]
    return domestic, international
```

근데 생각하면 할수록 이상했다.

애초에 수집 대상도 다르고, 기사 수도 다르고, 성격도 다른데 점수 하나로 합쳐버리면 나중에 설명을 못 하겠더라.

정규화도 고민해보고 언론사 개수 맞추는 것도 생각해봤는데...

솔직히 다 억지였다.

그래서 그냥 접었다.

국내 Top N이랑 해외 Top N을 따로 보여주고, 두 곳에서 동시에 나온 이슈만 🔗 표시를 붙이는 방향으로 결정했다.

괜히 욕심내는 것보다 이게 훨씬 깔끔했다.

(참고로 🔗 기능은 아직 없다.
설계만 끝났다.
언젠가는 만들겠지.)

그리고 아직도 리스트 컴프리헨션 한 줄에 길게 쓰는 건 적응이 안 된다.

## 네이버 수집기에 필요한 정보도 추가했다

같은 언론사가 같은 기사를 여러 번 내는 걸 걸러내려면 언론사 정보가 필요했다.

근데...

네이버 검색 API에는 언론사명이 없다.

저작권은 어디에 팔아먹었지.

```python
# naver_collector.py
def _extract_press(originallink: str) -> str:
    """
    네이버 API 응답에는 언론사명 필드가 따로 없다.
    originallink(원문 URL)의 도메인을 언론사 식별자로 대신 사용한다.

    예: "https://www.yna.co.kr/view/AKR2026..." -> "yna.co.kr"
    """
    if not originallink:
        return ""
    domain = urlparse(originallink).netloc
    domain = domain.replace("www.", "")
    return domain
```

잠깐 당황하긴 했는데, 원문 URL에서 도메인만 뽑아와서 press 필드를 만드는 걸로 해결했다.

description도 이제부터는 같이 저장하기로 했다.

```python
# naver_collector.py - search_naver_news() 반환 구조
results.append({
    "source": "네이버",
    "title": _strip_html_tags(item["title"]),
    "url": item["originallink"] or item["link"],
    "published_at": _parse_pub_date(item["pubDate"]),
    "category": None,   # 카테고리는 정제(normalizer) 단계에서 채움
    "body": None,       # 네이버는 본문을 안 줌
    "description": _strip_html_tags(item["description"]),
    "press": _extract_press(item["originallink"]),
})
```

그동안은 열심히 수집만 해놓고 거의 안 썼다.

이제 LLM 요약이라도 시킬 때 써먹어야지.

## 본문 추출 방식도 확정했다

`watt_collector`는 본문 컨테이너를 정확히 찾기보다, 추천 기사나 관련 기사 영역이 나오기 전까지의 `<p>` 태그만 수집하는 방식으로 구현했다.

```python
# WATT_collector.py - _fetch_detail()
paragraphs = []
for p_tag in soup.find_all("p"):
    text = p_tag.get_text(strip=True)
    if not text:
        continue
    if text.lower() in ("recommended", "related stories"):
        break
    paragraphs.append(text)
body = "\n\n".join(paragraphs) if paragraphs else None
```

## GDELT도 조금 더 알아봤다

공식 문서를 뒤져봤다.
(클로드가)

rate limit은 있다고 한다.

근데 숫자는 안 알려준다.

진짜 딱 이런 느낌이었다.

- 아저씨 여기 어떻게 가는지 아세요?
- 알아.

그래도 몇 가지는 확인할 수 있었다.

```python
# gdelt_collector.py

MAX_RETRIES = 4
BACKOFF_BASE_SECONDS = 60  # 60 -> 120 -> 240 -> 480초로 증가 (누적 900초)

# 일시적 네트워크 에러(ConnectTimeout 등) 재시도 횟수/대기시간.
# 429(rate limit)와 ConnectTimeout이 같은 실행 안에서 섞여 나오는 걸 확인함.
# 이건 요청 제한과는 별개로, 순간적인 접속 실패일 가능성이 높아 429보다는 훨씬 짧게 재시도.
NETWORK_ERROR_MAX_RETRIES = 2
NETWORK_ERROR_WAIT_SECONDS = 10
```

* 공식적으로 요청 제한이 존재한다는 점
* 실사용 사례에서는 짧은 시간에 요청이 몰리면 약 15분 정도 차단되는 경우가 있다는 점(공식은 아님)
* TIMESPAN 파라미터 형식
* ConnectTimeout은 429와 별개로 처리하는 편이 맞겠다는 점

공식 문서보다 남의 삽질 기록이 훨씬 도움이 됐다.

역시 인터넷 최고 스승은 누군가의 고통이다.
