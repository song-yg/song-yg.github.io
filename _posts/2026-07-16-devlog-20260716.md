## 출근하자마자 디스크가 터져 있었다

"No space left on device."

어제 봤던 그 문구를 오늘 또 봤다.

어제는 실행 도중 죽는 거였는데 오늘은 아예 시작부터 죽었다.

확인해보니 정리 전 여유 공간이 16GB(72GB 중 56GB 사용, 79%)였다.

GitHub-hosted 러너는 실행마다 완전히 새 VM이라 "여러 번 돌려서 누적된 것"은 아니라는 것도 확인했다.

범인은 `sentence-transformers`가 기본으로 끌어오는 GPU(CUDA)용 PyTorch였다.

BGE-M3(2.2GB), Playwright(`--with-deps`)까지 겹치니까 러너 여유공간을 그냥 넘어버렸다.

대응은 세 가지.

* 안 쓰는 `.NET`/`Android SDK`/`GHC` 같은 프리인스톨 툴 삭제
* PyTorch를 `--index-url https://download.pytorch.org/whl/cpu`로 먼저 설치해서 CUDA 휠 자체를 안 받게 함
* 단계별로 `df -h` 찍어서 어디서 얼마나 쓰는지 눈에 보이게 함

정리 후 35GB(19GB 확보).

이 정도면 살만하다.

`main.py` 실행 진입까지는 확인했다.

문제는 그 다음이었다.

## OpenRouter가 또 죽어 있었다

디스크를 고치고 나서 첫 실행.

3차 LLM 보조가 매번 `400 Client Error: Bad Request`로 실패했다.

로그를 보니 `model=`, 그러니까 빈 값으로 API를 부르고 있었다.

원인을 찾다 보니 리포 Variables에 `OPENROUTER_MODEL`을 등록을 안 해놨다.

근데 그럼 그냥 없는 거 아닌가?

아니었다.

GitHub Actions는 등록 안 된 Variable을 `${{ vars.X }}`로 참조하면 "아예 안 넘어옴"이 아니라 "빈 문자열로 채워서" 넘긴다.

공식 문서로 확인했다.

```python
# issue_grouper.py
# ** 2026-07-15 버그 수정 - os.environ.get(key, default) 대신 or 사용 **
# GitHub Actions에서 리포에 등록 안 된 Variable을 `${{ vars.X }}`로 참조하면
# "아예 안 넘어옴"이 아니라 "빈 문자열로 채워진 환경변수"가 된다(GitHub 공식
# 문서: "설정 안 된 configuration variable을 참조하면 빈 문자열로 평가됨").
# `os.environ.get(key, default)`의 default는 키가 "아예 없을 때"만 적용되고
# 빈 문자열이 있으면 그 빈 문자열을 그대로 돌려주므로, OPENROUTER_MODEL을
# Variables에 등록 안 한 상태로 workflow의 `OPENROUTER_MODEL: ${{ vars.OPENROUTER_MODEL }}`
# 를 그대로 두면 LLM_MODEL_OPENROUTER가 빈 문자열이 되어 OpenRouter API가
# "model" 필드 없음으로 400 Bad Request를 던지는 게 실제로 재현됨(실측
# 로그: "model=, 대상 66쌍" 다음 400 에러 26회). `or` 연산자를 쓰면 빈
# 문자열도 falsy라 기본값으로 자연스럽게 대체된다.
LLM_PROVIDER = os.environ.get("LLM_PROVIDER") or "anthropic"

LLM_MODEL_ANTHROPIC = "claude-haiku-4-5-20251001"
LLM_API_URL_ANTHROPIC = "https://api.anthropic.com/v1/messages"

LLM_MODEL_OPENROUTER = os.environ.get("OPENROUTER_MODEL") or "openrouter/free"
```

`or`로 바꾸니까 끝.

근데 코드 주석에는 "2026-07-15"라고 써있다.

날짜를 안 고치고 넘어갔다.

...이것도 나중에 누가(나겠지) 헷갈리겠다.

## 어제 고친 그 버그가 다시 나왔다

모델명 버그를 고치고 재실행.

이번엔 `UnicodeEncodeError`.

어? 이거 어제 고친 거 아닌가.

맞다.

근데 다시 났다.

```python
# issue_grouper.py - _call_llm()
if LLM_PROVIDER == "openrouter":
    headers = {
        "Authorization": f"Bearer {api_key}",
        "content-type": "application/json",
        # OpenRouter 권장 헤더(선택) - 프로젝트 식별용, 없어도 동작함.
        # 2026-07-16 수정: HTTP 헤더 값은 latin-1 인코딩만 허용되는데
        # 한글 문자열을 넣었다가 UnicodeEncodeError로 매 배치가 실패
        # 하는 게 실측으로 확인됨(3차가 시작은 됐지만 모든 배치가
        # 이 예외로 죽어서 결과적으로 이전 버그와 똑같이 0쌍 병합됨).
        # ASCII 전용 문자열로 교체.
        "X-Title": _OPENROUTER_X_TITLE,
    }
```

2026-07-15에 이미 한 번 고쳤던 것과 같은 계열(latin-1 인코딩) 문제가 형태를 바꿔서 재발한 거였다.

정확히 어디서 다시 한글이 섞여 들어갔는지는 이번 세션에서 못 짚었다.

대신 값 하나를 다시 ASCII로 고치는 걸로 끝내지 않고, 아예 인코딩 자체를 테스트로 검증하기로 했다.

값을 또 고쳐도 다음에 또 재발 안 한다는 보장이 없으니까.

```python
# issue_grouper.py
_OPENROUTER_X_TITLE = "feed-livestock-news-issue-grouping-stage3"
```

```python
# issue_grouper.py - 자체 점검 코드
# === OpenRouter 요청 헤더가 실제로 인코딩 가능한지 확인 (2026-07-16 추가) ===
# 배경: X-Title 헤더에 한글을 넣었다가 UnicodeEncodeError로 3차 LLM 보조의
# 모든 배치가 실패하는 게 실측으로 확인된 적 있음 (requests.post를 통째로
# mock으로 바꿔치기하는 위 스모크 테스트는 실제 HTTP 헤더 인코딩 단계를
# 건너뛰기 때문에 이 버그를 못 잡았음). HTTP 헤더 인코딩은 실제로는
# urllib3/http.client가 소켓에 쓰기 직전(더 아래 레이어)에서 수행하고
# requests.models.PreparedRequest.prepare_headers()는 값만 저장할 뿐 이
# 검증을 안 한다는 게 실제로 확인됨(테스트해봤더니 한글이 통과해버림) -
# 그래서 실제 실패와 동일한 지점인 str.encode("latin-1")을 직접 호출해서
# 검증한다. 앞으로 헤더에 비-ASCII 문자가 섞이면 여기서 바로 실패해야 한다.
for header_name, header_value in {
    "Authorization": "Bearer dummy-key-for-header-encoding-check",
    "content-type": "application/json",
    "X-Title": _OPENROUTER_X_TITLE,
}.items():
    header_value.encode("latin-1")  # 실패하면 UnicodeEncodeError로 여기서 바로 죽음
```

mock으로 `requests.post`를 통째로 바꿔치기하던 기존 테스트는 진짜 인코딩 단계를 건너뛰어서 이 버그를 못 잡았다.

`str.encode("latin-1")`을 직접 불러야 실제로 죽는 지점이랑 같아진다.

이건 왜 진작 안 했지.

## GDELT 429도 두 가지로 더 찔러봤다

여전히 심했다.

키워드마다 백오프 4단계를 다 소진하고 외부 재시도로 겨우 복구하는 게 반복됐다.

이번엔 두 가지를 같이 넣었다.

### User-Agent 헤더 주입

`gdeltdoc` 라이브러리 자체 이슈 트래커(#22)에서 "요청 수가 적어도 rate limit이 걸렸는데, User-Agent 헤더를 추가하니 해결됐다"는 보고를 다시 찾았다.

라이브러리에 이 패치가 병합됐다는 근거는 못 찾았다.

README에도 헤더 커스터마이징 옵션이 없다.

그래서 `requests` 기본 헤더 자체를 오버라이드하는 방식으로 우회했다.

```python
# gdelt_collector.py
_GDELT_USER_AGENT = (
    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 "
    "(KHTML, like Gecko) Chrome/126.0.0.0 Safari/537.36"
)

# 2026-07-16 버그 수정: requests.sessions 모듈이 `from .utils import
# default_headers`로 자기 네임스페이스에 직접 바인딩해서 쓰기 때문에,
# requests.utils.default_headers만 바꾸면 Session.__init__() 내부에서 부르는
# 건 여전히 원래 함수다 (실측 확인 - Session().headers["User-Agent"]가
# patch 후에도 그대로 "python-requests/x.y.z"였음). requests.sessions 쪽
# 바인딩도 같이 덮어써야 실제 Session 생성에 반영된다.
#
# 가드(_gdelt_ua_patched)를 두는 이유: 이 모듈이 어떤 경로로든 두 번 로드되면
# (예: importlib.reload) "원본 함수 캡처"까지 다시 실행돼서 이미 패치된
# 함수를 "원본"으로 잘못 캡처해버리고, 그걸 감싸는 새 wrapper가 자기 자신을
# 무한 호출하는 RecursionError가 실제로 재현됨(실측 확인) - 원본 캡처부터
# 최종 할당까지 전부 가드 안에 넣어서, 이미 패치돼 있으면 이 블록을 통째로
# 건너뛴다.
if not getattr(requests.utils, "_gdelt_ua_patched", False):
    _original_default_headers = _requests_utils.default_headers

    def _default_headers_with_gdelt_ua():
        headers = _original_default_headers()
        headers["User-Agent"] = _GDELT_USER_AGENT
        return headers

    requests.utils.default_headers = _default_headers_with_gdelt_ua
    requests.sessions.default_headers = _default_headers_with_gdelt_ua
    requests.utils._gdelt_ua_patched = True
```

패치 하나 넣으려다가 버그를 두 개나 만났다.

하나는 `requests.utils`만 바꾸면 실제 Session 생성에는 반영이 안 되는 것.

다른 하나는 모듈이 두 번 로드되면 이미 패치된 함수를 "원본"으로 착각해서 무한 재귀에 빠지는 것.

패치 하나에 방어막이 두 겹이 됐다.

효과는 아직 모른다.

### 키워드를 OR 하나로 합치기

`gdeltdoc`의 `Filters(keyword=[...])`에 리스트를 넘기면 자동으로 boolean OR로 묶인다는 걸 공식 README에서 확인했다.

기존엔 키워드 4개를 각각 호출해서 요청이 4번 나갔다.

```python
# gdelt_collector.py
def _collect_articles_for_keywords(gd: "GdeltDoc", keywords: list[str]) -> tuple[bool, list[dict]]:
    """
    여러 키워드를 하나의 OR 쿼리로 묶어서 article_search를 "한 번만" 호출한다.
    (2026-07-16 추가 - 위 모듈 docstring "429 대응 2건" 참고)

    gdeltdoc의 Filters(keyword=[...])는 리스트를 넘기면 값들을 boolean OR로
    묶어준다(공식 README 확인) - 기존엔 키워드 개수만큼 요청이 나갔는데
    이걸 1번으로 줄여서 429에 걸릴 기회 자체를 줄인다.

    ** MAX_RECORDS 상한이 이제 "키워드 조합 전체"에 걸린다는 점 주의 **:
    기존엔 키워드 하나당 최대 250건씩(키워드 4개면 이론상 최대 1000건)
    받을 수 있었는데, OR로 합치면 이 조합 전체에서 최대 250건까지만
    돌아온다. 지금 실측 키워드별 건수(합쳐도 250 미만)로는 문제가 없지만,
    나중에 키워드가 늘면 조용히 잘려나갈 수 있다.
    """
    label = " OR ".join(keywords)
    f = Filters(keyword=keywords, timespan=TIMESPAN, num_records=MAX_RECORDS)
```

시계열(`timelinevol`/`timelinevolraw`)은 "키워드별 트렌드"가 의미 있어야 해서 이건 합치지 않고 그대로 키워드별로 유지했다.

FALSE_POSITIVE 필터도 이제 어느 키워드가 매칭시켰는지 API가 안 알려주니까, 관련된 키워드 전부의 패턴을 모아서 검사하도록 바꿨다.

`collect()`의 외부 재시도 로직도 같이 손봤다.

기존엔 "실패한 키워드만" 골라서 다음 라운드에 재시도했는데, 이제 키워드가 하나의 조합으로 묶여서 요청이 나가니까 "실패한 키워드만"이라는 단위 자체가 없어졌다.

그래서 실패하면 조합 전체를 다시 재시도하는 방식으로 바꿨다.

적용 직후 실행(04시경)은 429가 1회로 확 줄었다.

오, 됐다 싶었다.

근데 1시간 뒤(05시경) 재실행에서는 429가 4회, 백오프 전량 소진까지 갔다.

표본 2회로 "대폭 개선"이라고 쓸 뻔했는데 정정했다.

키워드 결합으로 요청 횟수 자체가 4번에서 1번으로 준 건 확실한 사실이다.

근데 이게 429 발생 빈도를 실제로 줄여주는지는 아직 모른다.

## GDELT 공식 블로그를 더 팠다

이번엔 조금 다른 걸 찾았다.

GDELT 공식 블로그("Behind The Scenes: API Quotas & The Impact Of A Fraction Of A QPS", 2024-12-21)에 이런 내용이 있었다.

함대 전체(fleet-wide) QPS가 0.001만 늘어도 429 에러율이 5%까지 치솟는다.

그리고 가장 대규모 프로젝트들의 핵심 설계 목표는 quota를 넘지 않는 선에서 최대한 가깝게 도달하는 것이라고 한다.

즉 GDELT는 시스템 용량을 의도적으로 한계치 근처까지 채워서 운영한다.

이 말은 rate limit이 "나"라는 개별 클라이언트 단위가 아니라, 그 순간 전 세계 트래픽 총량 단위로 걸릴 수 있다는 뜻이다.

세션 중에 "우리 같은 작은 규모가 몇 명 접속해야 터지는 거냐"는 질문이 나왔었는데, 정리하고 보니 이건 "우리가 몇 명이냐"의 문제가 아니라 "그 순간 우리랑 무관한 전 세계 트래픽과 타이밍이 겹치느냐"의 문제에 더 가까웠다.

그렇다면 User-Agent도, 키워드 결합도 원천적으로 효과가 제한적일 수 있다.

근데 그렇다고 안 할 이유는 없다.

지금 구현된 백오프 + 외부 재시도가 이런 상황에는 맞는 대응이라는 쪽으로 무게가 실렸다.

## 4단계 LLM 요약을 붙였다

오늘 두 번째로 큰 작업이다.

`llm_summarizer.py`를 새로 만들었다.

(A) 그룹 내 제목(최대 10개)과 WATT 본문 발췌/네이버 description을 참고자료로 넘겨서 한국어 2~3문장 요약을 만든다.

(A-1) 그룹 크기가 1건이면, 즉 이슈 그룹핑이 안 된 단독 기사면 LLM 호출 자체를 생략하고 원문 제목만 노출한다.

```python
# llm_summarizer.py
def summarize_issue(item: dict) -> dict:
    """
    이슈 하나(scorer.score_group() 결과 dict)에 (A)/(A-1) 로직을 적용해
    요약을 붙인다. 원본 item은 변경하지 않고 얕은 복사본을 반환한다.
    """
    result = dict(item)
    titles = item.get("titles", [])

    # (A-1) 얇은 재료 fallback (문서 4번 섹션 (A-1), 2026-07-13 확정):
    # 이슈 그룹핑이 안 되고(그룹 크기 1) 언론사 1곳만 보도한 단독 기사는
    # 재료(제목 하나뿐)가 너무 얇아 LLM이 요약을 만들면 사실상 제목을
    # 부풀리는 것과 다름없으므로, 애초에 LLM 호출 자체를 생략한다.
    if len(titles) == 1:
        result["summary"] = None
        result["summary_skipped_reason"] = (
            "단독 기사(이슈 그룹핑 안 됨) - 재료가 얇아 요약 생략, 원문 제목만 노출 (문서 4번 섹션 (A-1))"
        )
        return result

    key_env_var = "OPENROUTER_API_KEY" if _ig.LLM_PROVIDER == "openrouter" else "ANTHROPIC_API_KEY"
    api_key = os.environ.get(key_env_var)
    if not api_key:
        result["summary"] = None
        result["summary_skipped_reason"] = (
            f"{key_env_var} 없음(LLM_PROVIDER={_ig.LLM_PROVIDER}) - 요약 생략, 원문 제목만 노출"
        )
        return result
```

프로바이더 설정(`LLM_PROVIDER`, 모델명, API URL, X-Title 상수)은 새로 만들지 않고 `issue_grouper.py`에서 그대로 가져다 썼다.

오늘 하루 저 설정값에서만 버그를 세 번 겪었는데, 여기서 또 베껴 쓰면 또 재발할 게 뻔했다.

이미 검증 끝난 값을 공유하는 게 안전했다.

진행상황 로그도 같이 넣었다.

```python
# llm_summarizer.py
def summarize_top_issues(ranked_items: list[dict], label: str = "") -> list[dict]:
    results = []
    total = len(ranked_items)
    for i, item in enumerate(ranked_items, start=1):
        titles = item.get("titles", [])
        rep_title = titles[0] if titles else "(제목 없음)"
        prefix = f"[llm_summarizer] {label} " if label else "[llm_summarizer] "
        print(f"{prefix}({i}/{total}) '{rep_title}' (그룹 {len(titles)}건) - 요약 요청 중...")

        result = summarize_issue(item)

        if result.get("summary"):
            print(f"{prefix}({i}/{total}) 요약 완료")
        else:
            print(f"{prefix}({i}/{total}) 요약 생략 - {result.get('summary_skipped_reason', '사유 불명')}")

        results.append(result)
    return results
```

이슈 하나당 LLM 호출이 몇 초에서 몇십 초까지 걸릴 수 있는데(특히 오픈라우터 무료 모델), 그 사이 로그가 조용하면 멈춘 건지 도는 건지 구분이 안 됐다.

GDELT/WATT collector에서 이미 쓰던 `(N/전체)` 패턴을 여기도 그대로 가져왔다.

`main.py`의 스텁 함수도 실제 로직으로 교체해서 연결했다.

이제 파이프라인이 수집 → 정규화 → 스코어링 → LLM 요약까지 이어진다.

## 로그가 늦게 뜨는 문제도 있었다

GitHub Actions 로그를 보는데 이상한 게 있었다.

HF Hub 경고(stderr)가 우리 자체 `print()`(stdout)보다 항상 먼저 떴다.

찾아보니 파이썬이 터미널이 아니라 로그 파이프 같은 곳에 출력할 때는 stdout이 기본적으로 블록 버퍼링된다고 한다.

실제로는 진행 중인데 로그에는 한참 아무것도 안 뜨다가 나중에 한꺼번에 쏟아지는 것처럼 보였던 이유가 이거였다.

workflow 실행 명령을 `python main.py`에서 `python -u main.py`로 바꿔서 버퍼링을 껐다.

별거 아닌 옵션 하나인데, 이거 하나로 로그 읽는 느낌이 완전히 달라졌다.

## 근데 마지막에 이상한 걸 하나 발견했다

시계열 수집을 껐다 켰다 하는 부분을 다시 들여다보다가 이상한 걸 봤다.

어제 기록에는 분명 `main.py`가 `gdelt_collector.collect(skip_timeline=True)`로 명시적으로 호출하도록 바꿨다고 돼 있었다.

근데 지금 실제 `main.py`의 `run_collectors()`는 그냥 `gdelt_collector.collect()`를 인자 없이 부르고 있었다.

`skip_timeline` 인자 자체가 안 보였다.

로그도 확인해봤다.

`[gdelt] 'foot and mouth disease' 시계열 수집 완료` 같은 줄이 실제로 찍혀 있었다.

즉 꺼놨던 게 다시 켜져 있었다.

언제 어떻게 되돌아갔는지는 이번 세션에서 못 찾았다.

저장 레이어가 아직 없어서 지금 켜져 있어도 그 데이터는 화면에도 안 뜨고 저장도 안 되고 그냥 버려지는 건 어제 판단이랑 같다.

근데 "분명 껐다고 기록했는데 코드는 켜져 있다"는 게 찜찜하다.

세션 중에 담당자가 여러 번 "지금 실제로 뭐가 돌고 있는 거냐"고 물었는데, 오래된 로그 근거를 새로 확인 안 하고 계속 우려먹다가 최신 상태랑 어긋난 답을 반복한 적이 있었다.

실행 상태를 물으면 무조건 제일 최근 로그부터 다시 봐야 한다는 걸 오늘 다시 배웠다.
