```python
# category_aggregator.py
def count_by_category(articles: list[dict]) -> Counter:
    """
    기사 리스트 하나를 받아 카테고리별 건수를 센다.
    articles는 keyword_tagger.tag_articles()가 이미 "category" 필드를 채워둔 상태여야 한다.
    """
    return Counter(a.get("category", "기타") for a in articles)


def aggregate(articles: list[dict]) -> dict[str, Counter]:
    """
    3.1 원칙대로 국내/해외 두 축으로 나눠서 각각 카테고리별 건수를 집계한다.
    반환값: {"국내": Counter, "해외": Counter}
    """
    domestic, international = scorer.split_domestic_international(articles)
    return {
        "국내": count_by_category(domestic),
        "해외": count_by_category(international),
    }
```

## GDELT랑 하루 종일 싸웠다

오늘의 MVP.

GDELT.

429가 26번 발생했다.

수집 하나 끝내는 데 78분.

처음에는 서버 죽은 줄 알았다.

그래서 하루 종일 GDELT 대응만 했다.

```python
# gdelt_collector.py
# 거의 매 호출마다 429가 발생해 재시도 백오프가 누적되는 문제가 있어 15초로 상향.
REQUEST_INTERVAL = 15.0

# 4단계(60→120→240→480초, 누적 900초=15분)로 늘려서 참고치와 누적 대기시간을 맞춤.
MAX_RETRIES = 4
BACKOFF_BASE_SECONDS = 60
```

요청 간격도 늘리고.

백오프도 늘리고.

실패한 키워드는 스킵시키고.

FALSE_POSITIVE도 넣고.

여기까지 하면 끝날 줄 알았다.

```python
# gdelt_collector.py
# "foot and mouth disease"(구제역) 검색 -> "hand, foot and mouth disease"
# (수족구병 - 어린이 질환, 전혀 다른 병)가 그대로 포함 매칭됨
FALSE_POSITIVE_FILTERS = {
    "foot and mouth disease": ["hand, foot and mouth", "hand foot and mouth"],
}
```

안 끝났다.

결국 실패한 키워드만 다시 돌리는 outer retry까지 만들었다.

```python
# gdelt_collector.py
# 총 시도 횟수 = 1(최초) + OUTER_RETRY_PASSES(추가 라운드) = 기본 3회.
OUTER_RETRY_PASSES = 2
OUTER_RETRY_WAIT_SECONDS = 90
```

429 발생 시각도 로그로 남기도록 했다.

(이 기능은 내일 다시 지우게 된다.)

## 이슈 그룹핑도 다시 검증했다

처음에는 질병명만 같으면 같은 그룹으로 묶어도 될 줄 알았다.

아니었다.

한국 조류독감/미국 조류독감/필리핀 조류독감...

전부 한 그룹으로 들어갔다.

이건 같은 질병이지, 같은 사건은 아니었다.

결국 1차 키워드 그룹은 비워두고 임베딩 중심으로 가기로 했다.

```python
# issue_grouper.py
# 재검증(calibrate_issue_grouper.py 실행) 결과.
# "조류독감" 묶음이 국내 기사 1건과 필리핀/캄보디아/호주/미국 등 전혀 다른 나라의 bird flu 기사를 전부 한 그룹으로 묶어버렸고,
# "구제역" 묶음도 한국 예천 발생 기사와 남아공 FMD 백신 관련 기사가 섞임.
# "완전 동일 사건" 매칭은 국가/장소/시점까지 구분해야 하는 훨씬 좁은 단위라 질병명 키워드 매칭으로는 표현이 애초에 불가능한듯.
ISSUE_SYNONYM_GROUPS: list[set[str]] = []
```

그리고 borderline 처리 순서에도 문제가 있다는 걸 발견했다.

이미 같은 그룹으로 묶인 기사들까지 LLM 후보로 올라가고 있었던 것이다.

```python
# issue_grouper.py - stage2_group()
# 1st pass에서 union만 먼저 전부 끝내고(간접 연결까지 확정)
for i, j in combinations(range(n), 2):
    sim = float(sim_matrix[i][j])
    if sim >= threshold:
        uf.union(i, j)

# 2nd pass에서 borderline 후보를 검사할 때 "두 기사가 이미 같은 그룹인가"를 확인해서, 이미 그룹이 확정된 쌍은 건너뛴다.
borderline_pairs = []
for i, j in combinations(range(n), 2):
    if uf.find(i) == uf.find(j):
        continue
    sim = float(sim_matrix[i][j])
    if threshold - borderline_margin <= sim < threshold:
        borderline_pairs.append((articles[i], articles[j], sim))
```

순서만 바꿨는데 borderline 후보가 446개에서 70개까지 줄어들었다.
